The Wikipedia API

Unlike the Google Trends interface we have worked with elsewhere, Wikipedia provides a fully official, documented, and free REST API — no scraping, no session cookies, no rate-limiting for reasonable use. But, as always, this doesn’t mean the code you write today won’t break unexpectedly.

The API is maintained by the Wikimedia Foundation and documented at wikimedia.org/api/rest_v1. It provides access to:

  • Page view counts: how many times a given article was viewed, per day
  • Edit activity: how many times a given article was edited, per day
  • Top articles: the most-viewed articles on a given Wikipedia edition
  • …and much more

We will access both page views and edit data via custom functions that call the API directly using {jsonlite}. The two functions — get_article_views() and get_article_edits() — are already loaded from functions.R, and both work the same way: they construct a URL, call the API, and return a tidy data frame. Reading their code is a good introduction to working with REST APIs in R.

Interpreting Wikipedia Data

Before diving in, it is worth being clear about what these two metrics actually measure:

Page views count the number of times an article page was loaded by a human user (bots are filtered out). A spike in page views typically indicates that something happened — a news event, a speech, a viral moment — that sent people to Wikipedia to find out more.

Edit counts count the number of times the article text was changed. Wikipedia articles about ongoing events are edited in near real-time as events unfold — sometimes within minutes of breaking news. High edit activity can indicate:

  • New information being added as events develop
  • Controversy or disputed facts (leading to “edit wars”)
  • Vandalism and its subsequent repair

Both metrics are raw counts, not normalised like Google Trends data. If you want to compare across language editions of different sizes, you will need to normalise by total project activity — something we will do in Part 3.

Part 1: Page Views — Trump Across the Globe

The article “Donald Trump” is one of the most-viewed political articles across virtually all language editions of Wikipedia, and — critically for our purposes — the article title is the same in almost every language. This makes it unusually easy to compare interest across countries.

Cross-National Map: Which Countries Are Reading About Trump?

Wikipedia is published in hundreds of languages, each with its own reader community. By summing page views across language editions and mapping them to their primary country, we get a rough proxy for cross-national interest.

This is an imperfect measure — English Wikipedia is read worldwide, not just in the US — but it is a useful approximation, and its limitations are themselves worth discussing.

Part 2: Edit Activity — Wikipedia as a Real-Time Record

If page views measure reading, edit counts measure writing. Every change made to a Wikipedia article is recorded — who made it, when, and what changed. For high-profile topics, this creates a rich time series of editorial activity that closely tracks real-world events.

The “Donald Trump” article on English Wikipedia is one of the most-edited political articles in Wikipedia’s history. Plotting daily edit counts reveals clear spikes around major events: court appearances, elections, executive orders, and controversies. This kind of analysis — linking edit surges to real-world moments — is an active area of research using Wikipedia as a data source.

Part 3: Extension — Normalising the Data

The Problem with Raw Counts

The maps in Parts 1 and 2 tell us how many times each language edition’s readers viewed or editors changed the Trump article. But this conflates two things:

  1. Relative interest in Trump within a given Wikipedia community
  2. The overall size of that Wikipedia edition

English Wikipedia is read by hundreds of millions of people worldwide; Vietnamese Wikipedia is much smaller. Even if Vietnamese readers were proportionally more interested in Trump, English Wikipedia would almost certainly show higher raw view counts. Raw counts tell us about absolute volume, not relative attention.

Normalising divides each article’s count by the total activity on that edition, giving us something closer to “what share of this edition’s readers were looking at Trump?” This is a more meaningful cross-national comparison.

Fetching Total Project Views

The get_project_views() function (loaded from functions.R) calls the Wikimedia aggregate pageviews endpoint — the project-level counterpart to get_article_views(). It returns total human pageviews for an entire Wikipedia edition per day. We sum this over the same 30-day window used above to get a denominator for normalisation.

project_views_list <- lapply(lang_country$language, function(lang) {
  Sys.sleep(0.25)
  tryCatch({
    pv <- get_project_views(project = paste0(lang, ".wikipedia"),
                            start   = start_date,
                            end     = end_date)
    data.frame(language = lang,
               total_project_views = sum(pv$views, na.rm = TRUE))
  }, error = function(e) {
    message("Could not fetch project views for: ", lang, " — ", e$message)
    data.frame(language = lang, total_project_views = NA_integer_)
  })
})

Normalising Edit Counts

We apply the same logic to edit counts. Here we normalise edits by total project views (not project edits) — this uses total readership as a proxy for edition size, which is a reasonable denominator when project-wide edit data is not readily available.

    group_by(language) %>%
    summarise(total_edits = sum(edits, na.rm = TRUE), .groups = "drop") %>%
    left_join(lang_country, by = "language")

  norm_edits_df <- lang_edits_totals %>%
    left_join(project_views_df, by = "language") %>%
    mutate(
      edits_per_million_views = total_edits / total_project_views * 1e6
    )

  norm_edits_df %>%
    select(language, country, total_edits, total_project_views, edits_per_million_views) %>%
    arrange(desc(edits_per_million_views))

Independent Task

Pick one or two Wikipedia articles on a topic or event that interests you and use get_article_views() to plot their daily page view time series.

Some starting points:

  • A current news story — find the Wikipedia article for a recent event and look for spikes. Do views surge the moment news breaks, or with a delay? How quickly do they fall off?
  • Two sides of a story — compare views for two related articles (e.g., a policy and its main opponent, or a conflict and its political resolution). Do they move together, or does one lag the other?
  • A recurring event — something with a predictable schedule (an election, a sporting event, a TV series). Does the pattern in the data match what you would expect?

For any article on English Wikipedia, the URL gives you the article name directly: https://en.wikipedia.org/wiki/ARTICLE_NAME. Pass that name to get_article_views(), replacing spaces with underscores.
